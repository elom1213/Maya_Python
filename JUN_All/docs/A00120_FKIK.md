# A00120_FKIK 사용법

## 1. 개요

캐릭터 리그의 **FK/IK 매칭 & 베이크**를 돕는 Maya 내 **PySide(Qt) 툴**이다. 선택한 리그
컨트롤에서 팔·다리(좌/우) 타깃/추종 컨트롤을 자동으로 모아, FK↔IK 포즈를 **스냅 매칭**하거나
선택한 **프레임 구간을 베이크**한다.

레거시 maya.cmds/MEL 툴(`PY_JUN_makeUI_FKIKTool`)을 현행 (B) Qt 아키텍처로 리팩토링한 버전이다.

- **로직/UI 분리**: 매칭·베이크 로직은 `app/core`(씬 접근만 `maya.cmds`), 화면은 `app/ui` 가 담당.
  core 는 위젯/컨트롤 이름을 모르고 `list[str]` 만 주고받는다.
- **UI**: PySide(`Framework.qt.qt` 가 PySide6→PySide2 폴백), Maya 메인 윈도우 자식으로 띄운다.
  Targets/Followers 리스트는 재사용 위젯 `Framework.qt.JUN_mod_tsl_qt`(A00140 와 동일).
- **Maya 2023 호환** (Python 3.9 / PySide2). UI 문자열/로그는 모두 영어.

## 3. 설치 / 실행

- **설치**: `A00120_FKIK/__dragDrop_A00120.py` 를 Maya 뷰포트로 **드래그&드롭**하면 현재 셸프에
  버튼이 설치된다.
- **실행**: 셸프 버튼, 또는 스크립트 에디터에서 `tools.A00120_FKIK.run(True)`(`True` 면 reload).

## 4. UI 사용 흐름

1. **Setup** — 리그 컨트롤을 선택하고 **Select Objects** → `FKIKSetup.resolve` 가 선택에서
   Targets/Followers 를 자동 해석해 두 리스트를 채운다. 상태 라벨(녹색=성공)로 결과 표시.
2. **Limb** — Arm Left/Right · Leg Left/Right 체크로 매칭·베이크 대상 사지를 한정한다(기본 전체 On).
3. **Targets / Followers** — 재사용 tsl 위젯(Select/Add/Del/Up/Down/Sort, 씬 선택 연동).
4. **Search** — token 입력 + (Invert) 로 두 리스트에서 일치 항목을 선택(씬에서도 select).
5. **Match / Bake**
   - **Start / End** : 베이크 구간. 각 칸 옆 **Get Current** 버튼으로 현재 Maya 프레임을 바로 넣을 수 있다.
   - **Match IK / Match FK** : 현재 프레임에서 한 번 포즈 스냅(`build_pairs` → `match`).
   - **Bake IK / Bake FK** : Start–End 구간을 **컨스트레인트 없이** 프레임마다 `matchTransform` 매칭 후
     키를 찍어 베이크한다(아래 §5).
   - **Bake (Constraint)** : 양쪽 리스트에서 **선택한 쌍**을 재생 범위 전체에 `parentConstraint` 로 걸어
     `bakeSimulation`(이름 그대로 의도적으로 컨스트레인트를 쓰는 별도 경로).
6. **Log / Force Refresh** — 작업 로그 표시. 뷰포트가 멈춘(커브 편집이 프레임 이동 전까지 안 보임)
   드문 경우 **Force Refresh** 로 막힌 전역 refresh suspend 를 풀고 한 번 다시 그린다.
7. **Help > About** — 창 맨 위 메뉴 바. 버전/날짜와 각 버튼 동작 요약을 보여준다.

## 5. 베이크 동작 상세

`FKIKMatcher.bake(tgt, flw, start, end)` 는 **컨스트레인트를 전혀 쓰지 않고** 구간(`[start, end]`)을
프레임마다 베이크한다. 각 프레임에서:

1. `currentTime(f)` 로 이동,
2. 각 (tgt, flw) 쌍을 `cmds.matchTransform(flw, tgt, position=True, rotation=True)` 로 매칭
   — follower 와 target 의 **rotateOrder 가 서로 달라도** 월드 포즈가 정확히 일치한다(A00145 Match 탭과 동일),
3. follower 의 `translate`/`rotate` 에만 그 프레임 키를 찍는다.

`setKeyframe` 은 **현재 활성 애니메이션 레이어**에 키를 쓰므로, 베이크 구간 밖의 키와 다른 레이어의
키는 전혀 건드리지 않는다. 대량 키 작업은 `suspend_refresh()` 로 감싸 프레임마다 리드로우하지 않는다.

### 왜 컨스트레인트 방식을 버렸나 (v01.07)

이전(v01.05~01.06)에는 임시 `parentConstraint` + `bakeResults` 로 베이크했는데, 컨트롤러가 **여러
애니메이션 레이어**(예: base + 추가 레이어)에 키를 가진 상태에서 특정 구간만 Bake 하면 **구간 밖의
포즈가 베이크 전과 달라지는** 문제가 있었다(원본 `JUN_PY_FKIK_Tool_V02_01` 에는 없던 현상). 원인은
`parentConstraint` 를 거는 순간 Maya 가 `pairBlend` 를 끼워 기존 `animCurve` 를 플러그에서 분리·머지하기
때문이며, 구간 밖 키를 스냅샷/복원하는 우회책도 **레이어를 인식하지 못해** 레이어 상황에서 실패했다.
컨스트레인트를 쓰지 않으면 이 문제가 원천적으로 사라진다.

> 예: 0~1000 프레임에 키가 있고 500~600 만 Bake → 0~500, 600~1000 의 원본 포즈가 그대로 남는다.

> **Bake (Constraint)** 버튼은 이름 그대로 의도적으로 `parentConstraint` + `bakeSimulation` 을 쓰는
> 별도 경로다(`bake_constraint()`). 이쪽은 재생 범위 밖 키에 스냅샷/복원 가드(`_snapshot_outside_keys` /
> `_restore_outside_keys`)를 그대로 유지한다.
